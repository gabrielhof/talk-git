História do Git


1991 Kernel do Linux é mantido através de arquivos compactados (tar.gz)
2002 O Kernel começa a ser mantido por um VCS distribuído e proprietário chamado BitKeeper
2005 Conflitos entre a comunidade Linux e a empresa dona do BitKeeper ropem a parceria entre os dois
2005 Nasce o Git
A comunidade opta por não usar o CVS na época. Falar dos problemas de lentidão do CVS. Falar do vídeo onde o Línus fala que arquivos compactados são melhores que o CVS. E falar da piada dele que o SVN "é o CVS feito de maneira certa" e que isso não é possível.

Repositório

O histórico dos arquivos é armazenado dentro do diretório .git
Não vou falar de como o Git armazena seus arquivos. Até porque eu nem sei direito. Mas a ideia da palestra é dar uma introdução e despertar o uso de Git. Não quero dar um passo maior que a perna.

Stage/Index

  • No Git, um arquivo pode ter três estados: Comitado, modificado e "staged" (preparado)

  • Um arquivo "staged" é um arquivo que tem sua versão atual armazenada em uma área do Git.

  • Arquivos na área de "Stage" são candidatos para serem armazenados no repositório.
Ao "stagiar" um arquivo, o Git armazena (de alguma forma) as modificações do arquivo. Se você alterar um arquivo já "stagiado", você precisa adicionar as novas modificações ao "stage", pq o Git já armazenou as alterações antigas no "stage". Usar a analogia de um time de futebol para explicar sobre os candidatos para o commit. Falar que é muito mais comum adicionar todos os arquivos ao "stage" do que um por um.

Stage/Index

Commit

  • Um commit é um objeto que aponta para as modificações realizadas nos arquivos do repositório.

  • Todo commit é identificado por um hash, possuí uma descrição e informações adicionais (como data e autor).

  • Todo commit possuí uma referência para o commit anteior.
Explicar por que é necessário informar o email e o nome para poder realizar um commit. Lembre-se disso: o commit possuí uma referência para o commit anterior.

Commit

Branching ou ramificação

  • Um branch é apenas um objeto que aponta para um commit.

  • Como todo commit possuí uma referência para o commit anterior, um branch acaba sendo uma referência para todo um fluxo de evolução do repositório.

Branching ou ramificação

  • Um repositório Git sempre está "focado" em um branch.

  • Quando você faz um commit, o branch atual passa a apontar para esse novo commit.
Explicar que o HEAD (que é tipo uma variavel) guarda qual o branch atual.

Branching ou ramificação

  • O branch padrão de um repositório Git é o master

  • Você pode criar branches para corrigir problemas, adicionar novos arquivos, testar, etc...

Pensem que você estão desenvolvendo uma nova funcionalidade. Aí do nada, liga aquele cliente chato, dizendo que o sistema da dando problema, tem navio parado e mimimi. Usando branches, você pode comitar as alterações da modificação que você estava fazendo e pode começar um branch novo para corrigir o bug.

Tags

  • Uma tag também é um objeto que aponta para um commit, assim como um branch, porém ela é imutável.

  • Tags normalmente são usadas para apontar para commits importantes, como uma nova versão.

O objetivo da tag é simplesmente esse: criar versões.
